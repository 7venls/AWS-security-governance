# AWS とはなにか？（責任の境界で理解する）

## このテーマを学ぶ意味

AWS は「安全なクラウド」ではなく、安全に設計できる IT 基盤を提供しているサービス。

実際に発生している多くのAWS関連事故は、AWS 自体の問題ではなく利用者側の設定ミスや統制不足によるものである。

そのため、AWSを学ぶ上では最初に「誰がどこまで責任を持つのか」を明確に理解する必要がある。

## AWS の役割（AWS責任）

AWS が責任を持つのは、主に以下の責任である。

・データセンターの物理セキュリティ

・ハードウェア（サーバ、ストレージ）

・基盤ネットワーク

・クラウド基盤の可用性

AWS は利用者がどのように設定、運用するかまでは保証しない。

## 利用者の責任（利用者設定領域）

AWS を利用する場合、以下は利用者責任となる。

・IAM（誰が何をできるか）

・ネットワークの公開・非公開設定

・データの暗号化有無

・ログの取得・監視

・セキュリティ設定全般

これらが不十分な場合、

AWS を利用していても情報漏えいや不正アクセスは発生する。

## オンプレミスとの違い

・オンプレミス環境では、  
IT インフラのすべてを自社で管理する必要がある。

・AWS では物理部分の管理は AWS に委ねられるが、  
責任が軽くなるわけではなく、責任の分担が変わるだけである。

## IaaS / PaaS / SaaS の責任の考え方

### IaaS（例：EC2）

AWS：物理設備、基盤インフラ

利用者：OS、設定、アプリケーション

⇒ 自由度が高い分、利用者責任も大きい。

### PaaS（例：RDS）

・AWS：OS、DBエンジン管理

・利用者：ユーザー管理、アクセス制御、ログ確認

⇒ マネージドでも「お任せ」ではない。

### SaaS

・AWS：サービス全体

・利用者：アカウント管理・利用ルール

⇒ 技術よりも運用、人の管理が重要。

## 監査・ガバナンス視点

監査では「AWSを使っているかどうか」ではなく、以下が確認される。

・設定や設計を誰が決めたか。

・権限設計は適切か。

・ログによる検知体制があるか。

「AWSだから安全」という説明は、監査では通用しない。

## 学んだ結論

・AWS 責任を肩代わりしない。

・セキュリティは利用者の設計と運用に依存する。
